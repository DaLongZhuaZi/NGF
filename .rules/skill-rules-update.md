# 技能：规则滚动更新

**适用场景**：开发者在开发过程中发现了新的、可复用的规律、模式或踩坑经验，希望将其沉淀回 `.rules/` 规则库，供后续开发和 LLM 参考。

**触发方式**：由开发者手动提起，例如：
- "请把这个页面的写法沉淀为规则"
- "把刚才修复这个问题的经验更新到规则库"
- "更新规则：XXX 应该用 YYY 方式实现"

**不会自动触发**，LLM 不应自行判断是否需要更新规则，必须等开发者明确指令。

---

## 1. 更新原则

### 1.1 值得沉淀的内容

- 项目中**多个页面/模块**都会遇到的通用问题及其解决方法
- **官方 API 废弃/迁移**的适配模式（已在项目中验证可行）
- **ArkTS 类型安全**的典型错误与正确写法
- **框架层能力的正确接入姿势**，包括导入路径、参数顺序、生命周期配对
- **布局/安全区/沉浸式**的坑及其规避方式

### 1.2 不值得沉淀的内容

- 只发生过一次的、针对某个具体文件的特殊修复
- 临时绕过方案（未找到根因，只是让问题暂时消失）
- 具体业务逻辑（如某个演示页的数据格式、特定字符串）
- 与框架架构无关的样式微调

### 1.3 沉淀原则

- **宁少勿滥**：只有真正可复用的规律才进规则库
- **具体可操作**：规则必须包含代码示例，不要只写"应该这样做"
- **不重复 AGENTS.md**：AGENTS.md 已有的通用约束不在此重复，规则库聚焦**具体技能和模式**
- **标注验证状态**：沉淀时注明"已在哪个文件/场景验证"

---

## 2. 更新流程

### 第一步：判断目标规则文件

| 要更新的内容 | 目标文件 |
|-------------|---------|
| HDS 页面结构、顶栏、安全区、光效配置 | `skill-hds-page-design.md` |
| 主题/语言/视效/握持感知的接入方式 | `skill-manager-apis.md` |
| 跨领域的通用规律、新增独立技能 | 新建 `skill-<名称>.md` + 更新 `README.md` |
| 本更新流程自身的改进 | `skill-rules-update.md`（本文件） |

### 第二步：定位插入位置

在目标规则文件中，按以下优先级确定插入位置：
1. 如果是对现有小节的补充 → 追加到该小节末尾
2. 如果是现有小节的纠正 → 直接替换错误内容，注明"已更新"
3. 如果是全新独立规律 → 在文件末尾添加新的 `## N. 标题` 小节

### 第三步：更新内容格式

每条新增规律按以下格式写入：

```markdown
### X.Y 规律/模式名称

**背景**：（一句话说明为什么需要这个规律）

**正确做法**：

\`\`\`typescript
// 经过验证的代码片段
\`\`\`

**错误做法**（可选，如果有典型对比）：

\`\`\`typescript
// 不要这样写
\`\`\`

**验证来源**：`pages/ngf/SomeFile.ets`（已在此文件中验证）
```

### 第四步：同步更新 README.md 与 AGENTS.md（如有新文件）

如果新增了独立的 `skill-*.md` 文件，必须执行以下同步操作：
1. 在 `.rules/README.md` 的“规则文件列表”中追加一行，登记新技能文件。
2. 在根目录 `AGENTS.md` 的 `1.1 技能规则库（.rules/）` 章节的触发条件表格中追加一行，定义新技能的触发条件。这一步至关重要，否则新添加的规则文件永远不会被 Agent 自动嗅探和触发。

---

## 3. 新增独立技能文件模板

当需要新建 `skill-<名称>.md` 时，使用以下模板：

```markdown
# 技能：<技能名称>

**适用场景**：（一句话描述在什么情况下需要阅读本文件）

**不会自动触发**，需要开发者/LLM 在任务开始前主动阅读。

---

## 1. 背景

（技能涉及的框架层、组件或 API 背景说明）

## 2. 标准模式

（核心接入模式，含完整代码示例）

## 3. 常见错误与正确写法

（对比表格或代码块）

## 4. 关键文件路径速查

| 文件 | 说明 |
|------|------|
| `path/to/file.ets` | 说明 |
```

---

## 4. 禁止行为

- **禁止**：LLM 在没有开发者明确指令的情况下自行修改规则库文件
- **禁止**：把只适用于单个文件的特殊处理写入规则
- **禁止**：把 AGENTS.md 中已有的通用规则复制到规则库（避免二义性）
- **禁止**：删除已有规律（除非该规律已被官方 API 废弃或被证明错误）

---

## 5. 更新后的自检清单

执行规则更新后，逐项确认：

- [ ] 新内容有完整、可运行的代码示例
- [ ] 代码示例中的导入路径是项目真实路径（不是伪路径）
- [ ] 标注了"验证来源"文件
- [ ] 没有在规则中硬编码单一业务名称或特定字符串
- [ ] 没有与 AGENTS.md 中已有规则产生矛盾
- [ ] 如果新增了规则文件，已更新 `.rules/README.md` 的列表，**并且已在 `AGENTS.md` 注册了触发条件**

---

## 6. 典型触发用语示例

开发者可用以下方式触发规则更新：

```
"请把刚才写的 HdsNavDestination 安全区处理方式沉淀进规则库"
"更新 skill-manager-apis.md，补充刚才发现的 @Watch 与 @StorageProp 联用时的注意事项"
"新建一个关于 LazyForEach 使用规范的技能文件"
"把今天修复的类型安全问题经验更新到规则库"
```

LLM 收到上述类型的指令后，先阅读目标规则文件的现有内容，再执行精准插入，最后向开发者确认已更新的位置和内容摘要。
